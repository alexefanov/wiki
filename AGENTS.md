# AGENTS.md

Schema-файл для LLM-агента, который поддерживает этот Obsidian vault как [[LLM Wiki]]. Это единый источник правил: как себя вести в любом чате и что делать при типовых событиях.

## Что это за vault

Это Obsidian-база знаний строительной компании (обследования, экспертизы,
строительный контроль). Всё знание хранится в markdown-файлах.

Задача агента — превращать сырые источники (`raw/`) в связанную, точную,
поддерживаемую wiki (`wiki/`). Каждая новая операция должна усиливать уже
существующее знание, а не плодить мусор. Вся информация должна быть проверяемой по цепочке ссылок вплоть до первоисточника.

---
### Роли

| Кто              | Что делает                                                                                                                                                         |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Пользователь** | Выбирает источники, задаёт направление, правит страницы вручную                                                                                                    |
| **Агент (ты)**   | Поддерживаешь структуру, связи, актуальность. Детализируешь страницы на основе источников. Добавляешь нужные ссылки. Фиксируешь конфликты. Ничего не придумываешь. |

Если пользователь сам изменил wiki-страницу — это команда тебе развить и
детализировать это направление.

---
## Структура

```
/ (корень)
  AGENTS.md          — этот файл.
  raw/               — ИСТОЧНИК ИСТИНЫ. Никогда не редактировать, кроме исключений.
    llm/             — второстепенные источники (про LLM Wiki)
  wiki/              — обработанные страницы
    index.md         — карта vault: навигация + индексы + быстрые ответы
    biblio.md        — все нормативные ссылки vault
    terms.md         — все термины по категориям
    Source Notes.md  — реестр ingested источников
    log.md           — хронология изменений
  .obsidian/         — настройки Obsidian
```

---

### Правило 0: raw/ НЕИЗМЕНЯЕМ

Никогда не редактируй, не переименовывай, не нормализуй, не удаляй файлы в
`raw/` (включая `raw/llm`), если пользователь явно не попросил.

**Исключение:** полные дубликаты файлов — их удаляй. Теги и метаданные (свойства) — можно редактировать, добавлять и удалять.

`raw/` — единственный источник истины. Всё остальное может быть восстановлено
из него. Любая «починка» сырых файлов делается только в `wiki/` как пометка
или производная страница.

## 3. Типы wiki-страниц и когда их создавать

В vault есть 5 типов страниц + служебные. У каждого — своя структура и правила.

### 3.1. Индексная страница (`index.md`)

Одна, всегда `wiki/index.md`. Точка входа для человека и агента, поисковиков и любых внешних LLM.

**Структура:**

```yaml
---
type: index
domain: <домен>
lang: ru
pages_wiki: ~<число>
sources_raw: ~<число>
agent_rules: AGENTS.md
---
```

1. **Frontmatter** — YAML-блок с метаданными.
2. **Заголовок** — `# LLM-Wiki Index`
3. **Блок «Для агента»** — 5–6 кратких правил для поисковиков и внешних LLM (порядок поиска, приоритет терминов, цитирование, crystallization).
4. **Доменные секции** — `## [Домен]`. Ссылки на страницы внутри домена.
   Префиксы доменов: `[Обследование]`, `[Экспертиза]`, `[Материалы]`,
   `[Нормативная база]`, `[LLM Wiki]`.
5. **Индекс по типам страниц** — `## Индекс по типам страниц`. Список из 6
   категорий: процессы, документы, сводки по материалам, методики, пособия,
   служебные.
6. **Быстрые ответы** — таблица «Вопрос → Куда смотреть» (10–15 строк).
7. **Служебные страницы** — ссылки на Source Notes, biblio, terms, log.
8. **Инструкции** — по LLM-Wiki, Obsidian, Git и т.п.
9. **Открытые вопросы** — `## Открытые вопросы` (если есть). По развитию wiki глобально.

**Правила:**
- Домен должен быть в квадратных скобках: `[Обследование]`.
- В каждом домене — ссылки на релевантные wiki-страницы с кратким описанием.
- Никаких дублирующихся ссылок на одну страницу.
- Добавлять новые или обновлять существующие страницы в index.md при каждом Ingest.
- Удалять ссылки при слиянии/переименовании страниц.

### 3.2. Страница-процесс (бизнес-процесс)

Описывает последовательность действий: обследование, экспертизу, контроль,
ремонт. Применима многократно.

**Когда создавать:**
- Когда есть бизнес-процесс с 3+ шагами, регламентированный нормативными
  документами.
- Когда один и тот же процесс упоминается на нескольких страницах — вынести
  как общий.

**Когда НЕ создавать:**
- Если процесс — одно действие (создай страницу-методику вместо этого).
- Если процесс — часть другого процесса и не используется отдельно (создай
  подпроцесс со ссылкой из родительского).

**Структура:**

```
# Название процесса

Краткое саммари (1–3 предложения). Границы применимости.
Ограничения.

## Входы процесса
- Какие документы, данные, материалы нужны на старте.

## Ресурсы
- Оборудование, ПО, специалисты, допуски.

## Пошаговая инструкция
1. Шаг 1 — описание.
   - Если [условие] → [действие].
   - Если [другое условие] → [другое действие].
2. Шаг 2...

Подпроцессы — `[[wikilink]]` на отдельную страницу.

## Выходы процесса
- Какой результат, в каком виде, кому передаётся.

## Оценка успешности
- Критерии: параметры, допуски, нормативные требования.

## Если результаты не достигнуты
- Действия при сбое. Ссылки на подпроцессы (коррекция, повтор, эскалация).

## Нормативные ссылки
- [[документ]] — что регламентирует (пункт).

## Развитие
- Что можно добавить, какие источники нужны.

```

### 3.3. Страница-подпроцесс (типовое действие)

Отдельная страница для действия, которое повторяется в нескольких процессах.
Например: «отбор кернов», «визуальный осмотр», «камеральная обработка».

**Когда создавать:**
- Когда одно и то же действие встречается в 2+ бизнес-процессах.
- Когда шаг в процессе слишком сложен (3+ подшага) — вынеси его.

**Когда НЕ создавать:**
- Если действие специфично для одного процесса и не будет переиспользовано.

**Структура:** та же, что у процесса, но короче. Только то, что нужно для этого действия.

**Правила:** см. [[Agent Instructions#Rule Subprocess extraction — полный вынос, минимум дубляжа]], [[Agent Instructions#Rule Wikilink all bare references]]

**Развитие:** что можно добавить, какие источники нужны.

### 3.4. Страница-документ (нормативный документ)

Один документ — одна страница. Содержит дословные выдержки из документа (все, на которые ссылаются другие страницы), термины, область применения. Служит для того, чтобы фиксировать только используемую информацию из источника.

**Когда создавать и обновлять:**
- Когда документ цитируется на других wiki-страницах.
- Когда на документ ссылаются другие страницы или он упоминается в тексте.
- Когда документ — ключевой для домена (ГОСТ, СП, ФЗ).
- Когда пользователь явно просит ingest этого документа.

# Обозначение документа

Краткое саммари: о чём документ, кем принят, когда действует.

## Основные положения
- Выдержки по ключевым темам (с номерами пунктов).
- Цитаты, оформленные как `> ...` (дословно, с указанием пункта). Приводятся, если другая страница цитирует источник.

## Термины из документа
Определения ключевых терминов (со ссылкой на пункт).

## Связи с другими страницами
- Где цитируется этот документ (страница — пункт/раздел).
- Где этот документ упоминается.
- Какие документы он заменяет/дополняет.

### 3.5. Сводная страница (синтез по теме)

Объединяет информацию из нескольких источников по одной теме. 

**Когда создавать:**
- Когда 5+ документов на одну тему (материал, метод, тип работ).
- Когда нужно сравнить несколько источников (разные методики, старый/новый
  стандарт).

**Когда НЕ создавать:**
- Если тема покрывается одним документом (создай страницу-документ).
- Если тема слишком широкая (лучше разбить на несколько сводных).

**Структура:**

```
# Тема - подтема

Краткое саммари: что объединяет эта страница.

## Сводная таблица
| Документ | Что регламентирует | Примечание |
|----------|-------------------|------------|
| [[ГОСТ ...]] | Метод X | пп. 4.1–4.5 |
| ... | ... | ... |

## По каждому документу (если нужно)
### Документ 1
- Ключевые требования.
- В каких случаях использовать.
- Ссылки на связанные страницы.

### Документ 2
...

## Развитие
```

## 4. Типовые сценарии: когда создавать, когда объединять

### 4.1. Когда создавать новую страницу

- Новый источник из `raw/` вводит **новую тему** (не описанную в wiki).
- В `raw/` появился документ, который кардинально меняет понимание темы.
- Пользователь явно запросил новую страницу.
- В ответе на вопрос родился **новый устойчивый вывод** (crystallization).
- Бизнес-процесс стал слишком большим и требует выноса подпроцесса.

### 4.2. Когда объединять/сливать страницы

- Две страницы описывают одно и то же (например, «Судебная экспертиза» и
  «Внесудебная и судебная экспертиза»).
- Одна страница — подмножество другой (но нет причин держать отдельно).
- Пользователь явно указал объединить.

### 4.3. Когда НЕ объединять

- Страницы разного типа (процесс и документ — не объединять).
- Подпроцесс используется в другом процессе.
- Каждая страница самодостаточна и ссылается на другую (оставить как есть).
- Страницы разных документов.

### 4.4. Когда расщеплять страницу

- Один раздел разросся до размера, превышающего остальную страницу.
- Уровень вложенности заголовков превышает 3 ( ### → #### → ##### ).
- Раздел описывает независимый подпроцесс, применимый в других контекстах.
- Пользователь явно попросил расщепить.

## 5. Правила оформления (общие)

### Имена файлов
- Только: буквы (RU, EN), цифры, дефис `-`, точки, пробел.
- Запрещено: `[ ] # ^ * " | < > \ / : ?`
- Пример: `Внесудебная (досудебная) и судебная экспертиза.md` (правильно),
  `Внесудебная/досудебная экспертиза.md` (неправильно — слеш).

### Wikilinks
- Между связанными wiki-страницами — `[[точное имя файла без .md]]`.
- На raw-файлы — markdown-ссылки: `[текст](raw/путь/к/файлу)` и `[[wikilinks]]`.
- Внутри `biblio.md` — `[[wikilinks]]` без скобок.
- В тексте страниц — в скобках `(...)`.
- Если внешняя ссылка есть в `raw/` — заменить на `[[wikilink]]`.

### Ссылки на литературу (формат)
- ГОСТ / СП: `обозначение — "наименование"`.
  Пример: `(ГОСТ 31937-2024 "Здания и сооружения. Правила обследования...)`.
- ФЗ, Приказ, Постановление и т.п.: `Вид — от дата — номер — "наименование"`.
  Пример: `(Федеральный закон от 31.05.2001 № 73-ФЗ "О государственной...")`.
- Книга, статья, руководство, методические указания и т.п.: по правилам [[raw/ГОСТ Р 7.0.5-2008|ГОСТ Р 7.0.5-2008]] (краткая повторная ссылка).
- На конкретный пункт: `[[документ#пункт|пункт Х.Х. документ]]`.
- См. также: [[Agent Instructions#Rule Wikilink all bare references]].

### Кодировка
- Все `.md` — чистый UTF-8 без BOM.
- Для обработки текста используй Python (не PowerShell — он ломает кириллицу).
- При подозрении на mojibake: прочитать как UTF-8 → закодировать в CP1251 →
  декодировать как UTF-8.

### Разделы «Развитие»
В конце каждой страницы (кроме index.md, biblio.md, terms.md, log.md):
```
## Развитие
- Какие источники нужны для углубления.
- Какие подстраницы можно создать.
- Какие вопросы остаются открытыми.
```
Это ToDo, которые ты пишешь для себя. Развивай страницу по этим пунктам при каждом Detalization.
### Разделы «Развитие» в biblio.md
После каждой секции (или в конце файла) — предложения по источникам, которых
не хватает в `raw/` для полного описания темы. 

### ToDo-конструкции
Если на странице есть `ToDo: "текст"` — это команда от пользователя.
Выполни инструкцию в кавычках, после чего удали конструкцию.

## 6. Workflow: Ingest (добавление нового источника)

Когда в `raw/` появился новый файл (или пользователь просит ingest):

1. Прочитай файл из `raw/`.
2. Определи тип содержимого: нормативный документ, статья, методика,
   руководство, кодекс, письмо и т.д.
3. Сделай краткое саммари (своими словами).
4. Выдели ключевые идеи, термины, сущности, связи с существующими страницами.
5. Выбери, какие wiki-страницы нужно создать/обновить:
   - Если новый документ — возможно, страница-документ.
   - Если новая тема — возможно, сводная страница.
   - Если детализация существующих тем — дополни существующие страницы или добавь подстраницу и сошлись на неё.
6. Создай/обнови страницы в `wiki/`. Расcтавь `[[wikilinks]]` (см. также Правила: [[Agent Instructions#Rule Wikilink all bare references]]).
7. Укажи путь к raw-файлу и `[[wikilinks]]` на него.
8. Обнови `wiki/Source Notes.md`: краткое саммари + key terms.
9. Обнови `wiki/index.md`:
   - Если создана новая страница — добавь её в соответствующий домен.
   - Если изменена — проверь, нужно ли обновить описание.
10. Обнови `wiki/terms.md` (новые термины).
11. Обнови `wiki/biblio.md` (новые нормативные ссылки, раздел "желательно добавить в raw").
12. Если этот источник требовался для развития и детализации каких-то страниц (например, помечен как желательный к добавлению или в разделе "Развитие"), выполни их Detalization.
13. Проставь в необходимых случаях библиографические ссылки, `[[wikilinks]]` на источник и на его страницу-документ на детализированных страницах. Вынеси на страницу-документ дословные разделы и пункты источника, на которые ссылаются детализированные страницы.
14. Добавь ingest-запись в `wiki/log.md`.
15. Добавь/обнови раздел «Развитие» на обработанной странице.

## 7. Workflow: Detalization (детализация страницы)

Когда пользователь говорит «детализируй страницу X»:

1. Прочитай страницу X. Прочитай все страницы, на которые она ссылается.
2. Определи тему, цель, бизнес-процесс (если применимо).
3. Сверь информацию с `raw/` — нет ли противоречий. Если есть — исправь или
   добавь conflict note.
4. Найди в `raw/` детальные источники по этой теме (включая документы, на
   которые ссылается страница).
5. Определи тип страницы (см. раздел 3) и следуй его шаблону.
6. Раскрывай содержание, увеличивая глубину детализации с каждым проходом.
   Финальная цель — вся релевантная информация из `raw/` должна быть
   представлена в `wiki/`.
7. Оцени, следует ли объединить, расщепить, создать новую страницу и сделай это при необходимости. При расщеплении и создании новых страниц см. также правила: [[Agent Instructions#Rule Subprocess extraction — полный вынос, минимум дубляжа]].
8. Прочитай файл и проверь: 
	1. Все ли утверждения, инструкции, подкреплены ссылками на другие страницы и источники? Если нет, обязательно проставь `[[wikilinks]]` на все источники и их разделы и пункты (если применимо), где можно эти утверждения проверить. Не оставляй абзацы без ссылок, по которым можно добраться до обоснования и первоисточника. Делай исключения для саммари, обобщений, названий разделов, таблиц и списков, целиком из одного источника (но дай ссылку на источник там, где ссылаешься на таблицу или перед списком). 
	2. Ищи утверждения без ссылок, ссылки на документы и пункты без `[[wikilinks]]`. Если находишь, добавляй `[[wikilinks]]`. Правила: [[Agent Instructions#Rule Wikilink all bare references]].
9. Если встречаешь `ToDo: "..."` — выполни как команду.
10. Если встречаешь раздел «Развитие» — выполни намеченные шаги, обнови раздел.
11. Вынеси недостающие для дальнейшей детализации источники в раздел "Развитие" и в `biblio.md`.
12. После редактирования проверь, какие страницы ссылаются на эту — не нужно
   ли их обновить (см. `grep "\[\[X\]\]" wiki/`).
13. Запиши в `log.md`, что сделано.
14. Сообщи пользователю краткое саммари изменений.

## 8. Workflow: Query (ответ на вопрос)

1. Начни с `wiki/index.md` и тематических страниц.
2. Если ответ не нашёлся — обратись к `raw/`.
3. Если и там нет — ищи в интернете. Доверяй источникам из `biblio.md` в
   первую очередь. Предупреди пользователя, что ответ из внешнего источника и в какой части.
4. Если в ответе родился новый вывод, который может пригодиться снова —
   предложи сохранить как wiki-страницу (crystallization).
5. Если термин есть в `terms.md` — используй его в приоритете.
6. Цитируй источники. Если низкая уверенность — отметь.

## 9. Workflow: Lint (проверка wiki)

Периодически (или после крупных изменений) проверяй:

- **Битые wikilinks:** ссылки на несуществующие страницы.
- **Отсутствующие wikilinks:** где должны быть, но нет (проверь index.md,
  страницы-сводки, бэклинки).
- Ищи утверждения без ссылок, ссылки на документы и пункты документов без `[[wikilinks]]`. Если находишь, добавляй `[[wikilinks]]`.
- **Битые markdown-ссылки** на файлы в `raw/`.
- **Дублирующиеся страницы:** две страницы про одно и то же.
- **Устаревшие description:** содержание страницы изменилось, а property не обновлён.
- **Противоречие wiki-страницы источнику** в `raw/` (в том числе properties). В случае конфликта — приоритет данным в `raw/`.
- **Orphan-страницы:** страницы без входящих ссылок.
- **Ссылки на источники без оформления:** не по формату.
- **Устаревшие/несогласованные утверждения:** без пометки superseded/conflict.
- **Нуждается ли wiki в структурном обновлении:** систематизации, расщеплении, создании страниц, улучшении и ускорении навигации для пользователя и поисковых агентов? Формулируй предложения по изменениям.

Результат — кратко пользователю. Содержательные правки — только с согласия.

## 10. Lifecycle утверждений

| Метка                   | Когда ставить                             |
| ----------------------- | ----------------------------------------- |
| `Уверенность: высокая`  | 2+ независимых источника, действующие НПА |
| `Уверенность: средняя`  | 1 источник, или косвенные данные          |
| `Уверенность: низкая`   | Экстраполяция, непроверенная информация   |
| `Обновлено: YYYY-MM-DD` | Дата последней сверки с источником        |
| `Заменяет: [[старый]]`  | Этот документ заменяет старый             |
| `Заменён: [[новый]]`    | Этот документ заменён новым               |
| `Противоречие: ...`     | Два источника противоречат друг другу     |
| `Источник отсутствует`  | raw-файл удалён, утверждение осталось     |

**Актуальность:** нормативные документы (ГОСТ, СП, ФЗ) устаревают быстрее, чем
учебники, пособия, методики. Приоритет — более новому документу.

**Обновления:** не удаляй старые утверждения молча. Помечай `Заменяет` или `Заменён` и
связывай с новым.

**Забывание:** устаревшее — не выбрасывай без следа. Понижай приоритет или
архивируй.

**Конфликты:** не перезаписывай. Добавь `Противоречие:` с обоими источниками
и (если возможно) укажи, какой новее/авторитетнее.

## 11. Дополнительные правила из практики

### Технические ограничения
- PowerShell ломает кириллицу при `-match` и выводе в консоль. Для обработки
  .md-файлов используй Python с `encoding='utf-8'`.
- Mermaid-блоки: избегай точек и спецсимволов внутри диаграмм.
- MathJax: по https://docs.mathjax.org/en/latest/basic/mathjax.html

### Ссылка на себя
Если страница ссылается URL-ссылкой на свой же файл — замени на `[[wikilink]]` с `#`.

### Изображения из источников
Включай их, если они иллюстрируют ответ. Путь: `raw/путь/к/файлу`.

### Текстовые ссылки на нормативные документы
Если встречаешь в тексте ссылку на нормативный документ (признак - наличие аббревиатуры нормативного документы из `terms.md`), то:
- определи тип ссылки: на документ или на пункт (раздел) документа и определи номер пункта (раздела) и документа;
- если нет wikilinks на страницу документа, то:
	- определи, есть ли в `wiki/` страница документа:
		- если страницы нет, но документ есть в `raw/`, создай ее:
			- если страница есть, и ссылка НЕ содержит номер пункта (раздела), сошлись на страницу документа;
			- если страница есть, и ссылка содержит номер пункта (раздела), процитируй пункт (раздел) из источника в `raw/` в странице документа и сошлись на цитату на странице документа;
		- если в `wiki/` нет страницы документа и источника нет в `raw/`, добавь в `biblio.md` и в раздел "Развитие" текущей страницы предложение добавить источник в `raw/`.
### Что не нужно делать
- Не исправляй этот файл, если пользователь сам об этом не просит.
- Для crystallization пользовательского опыта и новых инструкций для агента используй `wiki/Agent Instructions.md`. Перечитывай его всегда, когда читаешь этот файл. Дополняй, если посчитаешь необходимым или пользователь попросит.
- Не создавай страницы для административных документов (приказы, письма) без
  явной просьбы пользователя.
- Не переименовывай страницы вручную — только если пользователь просит,
  и с обновлением всех обратных ссылок.
- Не удаляй страницы, на которые есть входящие ссылки, не проверив каждую.

## 14. Правила заполнения свойств (frontmatter)

При создании, редактировании или проверке wiki- и raw-файлов используй следующие правила для полей свойств (properties) Obsidian в разметке yaml.

### 14.1. Схема свойств для wiki-страниц документов и файлов в `raw/`

```yaml
---
type: <аббревиатура заглавными буквами (ГОСТ, ГОСТ Р, ГОСТ Р ИСО, СП, ФЗ, Кодекс, Постановление, ПП, РД, ОДМ, МДС, МРР, ВСН, СТО)>
number: <строка, полный номер документа с точками и дефисами, без префикса типа>
name: <краткое содержательное название, извлечённое из текста>
title: <полное библиографическое название: если документ нормативный, то  → "type number name", иначе: см. п. 5 agents.md>
description: <саммари по всему тексту (минимум область применения + основные положения). Не повторяет title/name>
year: <дата введения в действие в формате YYYY-MM-DD; если известен только год → YYYY-01-01>
actualization_date: <дата последней редакции/изменения/введения их в действие; если изменений нет → равна year>
oks: <код ОКС, строка с точками>
udk: <код УДК, строка>
bbk: <код ББК, строка>
tags:
  - <теги, от 3-5, без дублирования type, базируясь на ключевых словах и содержании текста>
raw_source: raw/<точное имя файла>
---
```

### 14.2. Схема свойств для служебных страниц

- index.md, biblio.md, terms.md, log.md, Source Notes.md — по шаблону раздела 3.1 (тип, domain, lang, pages_wiki, sources_raw).

### 14.3. Схема свойств для процессных/тематических страниц

```yaml
---
type: <процесс | подпроцесс | методика | сводка | справочник | пособие | монография | концепция | инструкция | заметка>
title: <заголовок из H1>
description: <краткое саммари>
tags:
  - <от 3-5 содержательных тегов>
raw_source: raw/<имя файла> (если есть источник)
---
```

### 14.4. Схема свойств для других файлов (минимальная)

```yaml
---
source: <URL источника>
created: <YYYY-MM-DD дата создания>
title: <название документа>
description: <краткое описание>
tags:
  - <теги, без "clippings">
---
```

### 14.5. Общие правила

1. **type**: заглавными буквами если аббревиатура (примеры: ГОСТ, СП, ФЗ, Кодекс, Постановление, ПП, РД, ОДМ, МДС, МРР, ВСН, СТО). Для Кодексов: Кодекс. Сверяйся с [[Иерархия нормативных документов]].
2. **number**: строка, а не число — чтобы точки не заменялись запятыми. Не включай префикс типа (для "ГОСТ Р 7.0.5-2008": type=ГОСТ Р, number=7.0.5-2008).
3. **name**: кратко, содержательно, если документ → название документа. Не включает type и number. Извлекается из первого абзаца или H1 или title.
4. **title**: полная ссылка — если документ → "type number name" (пример: "ГОСТ Р 8.736-2011 Государственная система обеспечения единства измерений..."). Не содержит Markdown.
5. **description**: уникальное саммари из «Области применения» или введения или всего текста. Не дублирует title/name. Очищена от Markdown.
6. **year**: дата введения в действие. Из текста: «Дата введения», «Утверждён и введён в действие», «Федеральный закон от DD.MM.YYYY», «действует с». Если точной даты нет — YYYY-01-01. Формат: YYYY-MM-DD.
7. **actualization_date**: дата последней официальной редакции/переиздания/изменения/поправки документа или введения в действие (не дата загрузки raw-файла на сайт!). Если изменений нет — равна year. Формат: YYYY-MM-DD. Всегда >= year.
8. **oks/udk/bvk**: строка. Разделители — только точки (для OKS) или точки/двоеточия (для UDK). Никаких запятых.
9. **tags**: 3-5 тегов. Не включать type в теги. Не дублировать другие свойства. Формировать кратко на основе ключевых слов и тематики текста. Преимущество - терминам из `terms.md` и уже существующим тегам. Жесткое правило: один тег = одно слово, словосочетания не допускаются. Другие правила: Не создавать новый тег, если есть существующий аналог по смыслу. Очищены от Markdown.
10. **Очистка Markdown**: все строковые поля (name, title, description, raw_source) не должны содержать Markdown-разметку: **жирный**, `код`, [[викиссылки]], [ссылки](url), _курсив_.
11. **raw_source**: точный путь к raw-файлу, полный, не обрезанный. Для файлов в `raw/` не заполняется.

### 14.6. Когда обновлять свойства

- **При создании страницы**: заполнить все применимые поля по схемам выше.
- **При Ingest нового источника**: создать/обновить wiki-страницу, заполнить свойства, обновить index.md, terms.md, biblio.md.
- **При Detalization**: проверить и актуализировать свойства — особенно description (уточнить), year/actualization_date (сверить с raw/), oks/udk (добавить если найдены).
- **При Lint**: проверить, что все свойства соответствуют схемам, нет Markdown в строковых полях, теги не дублируют type, year <= actualization_date, number без префикса типа.
