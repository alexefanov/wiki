### Настройка Web Clipper

Выполните эти шаги, чтобы создать шаблон, который будет направлять заметки прямо в папку `raw`.

1. **Откройте настройки расширения**: Кликните по иконке Obsidian Web Clipper в браузере, затем нажмите на шестеренку (⚙️) в правом верхнем углу.
    
2. **Перейдите к шаблонам**: В открывшемся окне выберите вкладку **Templates**.
    
3. **Создайте новый шаблон**: Нажмите на синюю кнопку **New Template** или просто отредактируйте шаблон по умолчанию.
    
4. **Настройте параметры шаблона**: Вас встретят поля, которые определяют, как будет работать ваш шаблон. Заполните их:
    
    - **Name**: Дайте шаблону понятное имя, например, `Raw Web Clipper`.
        
    - **Note location** (Folder): В это поле **обязательно** впишите `raw`. Именно здесь будет указан путь для сохранения ваших заметок.
        
    - **Vault**: Выберите ваше хранилище (vault) из выпадающего списка.
        
    - **Note Name Format**: Здесь можно задать удобный формат имени файла. Например: `{{title}}` - по заголовку страницы.
        
    - **Note Content**: В этом большом поле настраивается содержимое самой заметки. Оставьте значение по умолчанию или вставьте свой вариант.
        
    - **Properties**: Здесь можно добавить метаданные в YAML-формате, такие как `source:`, `tags:`[](https://deepwiki.com/obsidianmd/obsidian-clipper/5.1-template-system).
        
    - **Triggers**: Оставьте это поле пустым, если шаблон должен быть доступен всегда, или укажите, например, `*`, чтобы он работал на всех сайтах. Можно добавить конкретные адреса для автоматического выбора шаблона, например, `https://medium.com/*`.
        
5. **Сохраните шаблон**: Нажмите кнопку **Save Template** внизу страницы. 
### Сохранение новых файлов и картинок в нужную папку

**Проблема:** 
- Новые страницы создаются в корне, а не в `wiki/`
- Web Clipper сохраняет `.md`-файл в папку `raw`, но картинки по умолчанию попадают в основное хранилище.

**Решение:**

1. **Укажите глобальную папку для вложений**  
   `Настройки Obsidian` → `Файлы и ссылки` → `Папка для новых вложений` → `В подпапке в текущей папке`.  
   *Картинки будут скачиваться туда, а не рядом с заметкой.*

2. **После сохранения заметки через Web Clipper** откройте её в Obsidian.

3. **Выполните команду** `Загрузить вложения для текущего файла`  
   - Вызовите палитру команд: `Ctrl/Cmd + P`  
   - Начните вводить `Загрузить вложения` → выберите нужную команду.  
   - Obsidian сам скачает все картинки по ссылкам и положит их в указанную на шаге 1 папку, а пути в заметке исправит.

4. **Для ускорения** назначьте этой команде горячую клавишу в `Настройки` → `Горячие клавиши` → найдите `Загрузить вложения для текущего файла`.
5. **Более верный вариант**: установить плагин Local Images Plus. Он сам будет скачивать файлы картинок.

### Git: ошибка `broken pipe` при stage / commit / sync и предупреждения в Obsidian Git  `LF will be replaced by CRLF`

**Проблема 1.** `error writing "file...": broken pipe`

**Причина:** слишком маленький буфер отправки Git, конфликт переводов строк (`CRLF`/`LF`) или заблокированный индекс.

#### Шаг 1. Увеличить буфер Git (решает 90% случаев)

Откройте **командную строку (CMD)** и выполните:

```cmd
git config --global http.postBuffer 524288000
```

Проверьте:

```cmd
git config --global --list
```

Должна появиться строка `http.postbuffer=524288000`.

#### Шаг 2. Отключить автоматическое преобразование переводов строк

```cmd
git config --global core.autocrlf false
```

Это избавит от предупреждений `LF will be replaced by CRLF` и возможных конфликтов.

#### Шаг 3. Удалить файл блокировки `index.lock` (если есть)

Перейдите в папку вашего хранилища, например:

```cmd
cd C:\Users\yefan.LAPTOP-T0BIR3IU\Documents\База
```

Удалите временный файл:

```cmd
del .git\index.lock
```

Если файла нет – пропустите.

#### Шаг 4. Перестроить индекс Git

В папке хранилища выполните:

```cmd
git reset
```

Эта команда заново прочитает все файлы и пересоздаст индекс. Ваши заметки не пострадают.

#### Шаг 5. Попробовать снова сделать stage / commit

После этих шагов ошибка `broken pipe` должна исчезнуть.

**Проблема 2.** `error: open("...Filename too long")` при `git add .`

**Причина:** ограничение Windows на длину пути (260 символов). Git в Obsidian натыкается на файл с очень длинным именем.

#### Решение: включить поддержку длинных путей

1. **В Git** (команда от имени администратора или просто глобально):

   ```cmd
   git config --system core.longpaths true
   ```

   Если нет прав администратора:

   ```cmd
   git config --global core.longpaths true
   ```

2. **В Windows 10/11** (требуется перезагрузка):

   - **Вариант А – через редактор реестра:**  
     `Win+R` → `regedit` → перейдите в  
     `HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\FileSystem`  
     Найдите параметр `LongPathsEnabled` → измените значение на `1`.  
     Если параметра нет – создайте `DWORD (32 бита)` с именем `LongPathsEnabled` и значением `1`.

   - **Вариант Б – через групповые политики (Windows Pro/Enterprise):**  
     `Win+R` → `gpedit.msc` → `Конфигурация компьютера` → `Административные шаблоны` → `Система` → `Файловая система` → `Включить поддержку длинных путей Win32` → `Включено`.

   - **После изменения реестра или политик перезагрузите компьютер.**

3. **После перезагрузки** снова выполните в папке хранилища:

   ```cmd
   git add .
   ```

   Ошибка `Filename too long` больше не появится.
